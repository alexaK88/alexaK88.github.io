Overview

This project simulates (or integrates with) a spacecraft telemetry stream and builds a complete monitoring flow: telemetry ingestion ‚Üí validation ‚Üí transformation ‚Üí storage ‚Üí health checks ‚Üí dashboards/alerts.

The focus is on reliability (schema enforcement, error handling, observability), performance (chunked processing), and operational usability (clear health indicators, trends, and diagnostics).

Problem & Goals

  • Problem: Telemetry is high-frequency and noisy. Without strict validation and health logic, issues can be missed (or false alarms become overwhelming).
  • Goal: Provide a robust pipeline that converts raw telemetry into reliable health insights: thresholds, derived metrics, event flags, and operator-friendly dashboards.
  • Outcome: A modular system that can be extended with new subsystems, new telemetry fields, and additional health rules with minimal changes.

System Architecture

The system is designed as containerized services for separation of concerns and easy local deployment. Typical components:

  • API Service (FastAPI): receives telemetry, validates payloads, stores/streams processed outputs.
  • Dashboard (Streamlit): real-time charts and subsystem health panels.
  • Observability: metrics endpoints + optional Prometheus/Grafana for system health.
  • Storage: file-based (CSV/JSON) or pluggable DB layer (optional extension).

Deployment: Docker + Docker Compose (multi-service orchestration).

Telemetry Data Flow

  1. Ingest telemetry packets/events (HTTP endpoint or simulated generator).
  2. Validate structure and types (schema enforcement, required fields, bounds).
  3. Transform (unit normalization, timestamp alignment, derived features).
  4. Health Rules (thresholds, rate-of-change checks, missing data detection).
  5. Persist processed stream for analysis & replay.
  6. Visualize subsystem panels + time-series trends.

Schema & Validation

Telemetry processing starts with strict schema validation to prevent silent failures and to keep downstream logic reliable.

  • Typed payloads (e.g., Pydantic models) for consistent decoding.
  • Constraints (range checks, enumerations, required fields).
  • Error reporting (invalid packets are rejected or quarantined, with clear logs/metrics).

Example fields you might include: timestamps, subsystem voltages/currents, temperatures, link status, memory usage, CPU load, attitude/orbit values, etc.

Health Monitoring Logic

Health monitoring converts raw telemetry into actionable signals:

  • Threshold checks: warn/critical limits per sensor (e.g., temperature, voltage).
  • Trend checks: detect drift (rate of change) and instability (spikes).
  • Missing data detection: identify stale sensors or transmission gaps.
  • Derived metrics: rolling averages, deltas, composite ‚Äúhealth score‚Äù.
  • Event stream: emit health events for dashboards and alerting.

(Optional upgrade) Add hysteresis + debounce to reduce alert flapping.

Dashboard & Operator View

  • Subsystem health panels (OK / Warning / Critical + details).
  • Time-series charts for key sensors and derived metrics.
  • Event timeline for alerts and anomalies.
  • Filters by subsystem, sensor, time window.

Observability

Besides spacecraft health, the system tracks its own health and performance:

  • Request rate, success/failure counts
  • Processing latency (ingest ‚Üí validated ‚Üí stored)
  • CPU/memory usage (service-level)
  • Error categories (validation vs runtime)

This makes the pipeline production-like and easier to debug under real streaming conditions.

Tech Stack

  • Backend: Python, FastAPI, Pydantic
  • Data: Pandas (processing), file-based storage (CSV/JSON) or pluggable DB
  • Frontend: Streamlit dashboards
  • DevOps: Docker, Docker Compose
  • Monitoring (optional): Prometheus, Grafana

Future Improvements

  • Streaming transport: WebSockets / MQTT / Kafka for higher-throughput ingestion
  • Rule engine: declarative health rules (YAML/JSON) per subsystem
  • Alerting: alert routing (email/Slack) + suppression/hysteresis
  • Replay mode: re-run historical telemetry for analysis and testing
  • ML layer: anomaly detection and predictive maintenance signals
